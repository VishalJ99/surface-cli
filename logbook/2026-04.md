## 2026-04-21 — OpenRouter Summary Backend Probing (PER-167)

**Objective:** Determine whether OpenRouter should be preferred over OpenClaw for
Surface summaries, and identify the best near-term concurrency and reasoning settings.

**Method:** Used live read-only `uni` mailbox fetch/search commands with summarization
disabled to freeze representative thread sets, then benchmarked OpenRouter models on
the same canonical summary payloads. Compared singleton, batched, and concurrent
single-thread calls. Used a read-only subagent quality pass over frozen source/output
JSON in `/tmp`.

**Observations:** `google/gemini-3-flash-preview` and `openai/gpt-5.2` were both much
faster than the current OpenClaw path on five-thread summary workloads. Qwen 3.5 397B
returned valid JSON in raw API probing but was too slow for the current timeout and
was dropped. Corrected quality analysis found all Gemini/GPT direct-batch outputs
aligned to the frozen thread refs. Gemini was faithful but terse. `gpt-5.2` with
`reasoning.effort = "none"` had the best balance of detail and action classification
on the sample. OpenRouter parallel single-thread calls scaled well up to five
concurrent calls on the small top-five workload. Large real university threads still
fit comfortably under a 64k-token-style budget after Surface clipping; the largest
sampled thread had 90 messages and an estimated 11.6k-token summary payload.

**Interpretation:** OpenRouter should be the next preferred summary path to implement,
with OpenClaw retained as fallback. For OpenRouter, prefer bounded parallel
single-thread calls first because they avoid cross-thread contamination and are already
fast; batching remains useful when caller overhead dominates but needs stricter
alignment checks. Reasoning should be configurable per model. For `gpt-5.2`, `none`
looked faster and better on this sample; for Gemini, reasoning controls were nearly
latency-neutral.

**Next:** Implement cascaded summarizer backends in PER-171 and keep PER-167 focused
on the current summary-latency evidence. Add OpenRouter fallback/parallelism with
forced-failure validation before changing the default.

## 2026-04-21 — GPT Summary Model Evaluation (PER-172)

**Objective:** Compare GPT-5.2 chat, GPT-5.3 chat, GPT-5.4 mini, and GPT-5.4 nano
for Surface thread summarization under the intended ME-scoped `needs_action`
semantics and causal-chain summary requirement.

**Method:** see `outputs/per172-gpt-summary-eval-20260421-113233/reproduction.txt`.
The run used 11 cached `uni` threads spanning single-message, self-latest,
external-latest, explicit action-required, informational, and long technical
coordination cases. Candidate models were called through the direct OpenAI API, and
`gpt-5.4` evaluated first-run outputs against the raw normalized thread payloads.

**Observations:** All four candidates returned valid JSON on every first-run thread.
Median first-run latency was `3.96s` for `gpt-5.2-chat-latest`, `3.25s` for
`gpt-5.3-chat-latest`, `1.46s` for `gpt-5.4-mini`, and `1.72s` for
`gpt-5.4-nano`. The evaluator scored `gpt-5.4-mini` and `gpt-5.3-chat-latest` at
`11/11` for ME-scoped action correctness; `gpt-5.2-chat-latest` and `gpt-5.4-nano`
were `10/11`. `gpt-5.4-nano` had the most reliability issues, including structured
action-flag inconsistency on repeated edge-case runs and three medium hallucination
risk flags.

**Interpretation:** On this first model-assisted benchmark, `gpt-5.4-mini` is the
best default candidate because it matched `gpt-5.3-chat-latest` on action accuracy
while being materially faster and having no medium/high hallucination flags.
`gpt-5.3-chat-latest` remains the quality fallback candidate. The benchmark also
surfaced a production gap: the `uni` account registry has a placeholder email, so
ME-scoped summaries need reliable authenticated account-owner identity before the
new action semantics are shipped.

**Next:** Update the summarizer contract/prompt/fingerprint around ME-scoped
`needs_action`, causal-chain briefs, and account-owner identity before changing the
default model.

## 2026-04-21 — Parallel Read-Path Probe (PER-175)

**Objective:** Check whether agent-style parallel CLI reads are safe enough to document before
shipping the summarization/session/identity release.

**Method:** Ran live read-only searches with `SURFACE_SUMMARIZER_BACKEND=none` against authenticated
`personal_2` Gmail and `uni` Outlook accounts. Probed two parallel Gmail searches, two parallel
cold Outlook searches, cross-provider parallel search, two concurrently started Outlook warm
sessions used in parallel, and two concurrent searches sharing one warm session.

**Observations:** All tested commands returned valid JSON and were cleaned up successfully. Gmail
same-account parallel search completed in about `1s`; cold Outlook same-account and cross-provider
parallel searches completed in about `12-13s`; two distinct warm sessions searched in parallel in
about `11s`; two commands sharing one warm session succeeded but took about `20s`.

**Interpretation:** Parallel read-only CLI use is acceptable for the skill, but Outlook concurrency
should be kept modest because browser-backed sessions contend for local resources. For planned
parallel Outlook work, one warm session per concurrent worker is the better default than sharing a
single session.

**Next:** Document this in the Surface skill and keep release validation focused on build/package
and publish checks.
